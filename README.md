# PS190 Firmware Reconstruction

This repository is a reconstructed C source tree for the PS190 HDMI 2.1 FRL retimer/repeater firmware.

It is not the original vendor source code.

The code here was rebuilt from a Hex-Rays decompilation of the firmware image and then rewritten into structured, human-readable C. The goal is behavioral fidelity to the original ARM Cortex-M4 bare-metal firmware, not just "something that compiles."

## What PS190 appears to be

- HDMI 2.1 FRL retimer / repeater device
- ARM Cortex-M4, Thumb-2, bare-metal firmware
- Handles video, FRL link training, eARC / ARC audio, SCDC / DDC, HDCP, flash / NVRAM, CEC, and assorted platform control

## What is in this tree

- `include/defs.h`
  - common types, memory-mapped register macros, shared declarations, and platform helpers
- `system_main.c`
  - reset / entry flow, exception handlers, scheduler primitives, main loop, PRNG
- `string_utils.c`
  - memory / string helpers and small low-level utility routines
- `libc_printf.c`
  - custom printf / vprintf implementation used by the firmware
- `softfloat.c`
  - software floating-point helpers used by the reconstructed firmware
- `flash_nvram.c`
  - SPI flash access, firmware image handling, NVRAM-style persistent storage logic
- `crypto_hdcp.c`
  - IMG4 / IM4M parsing, HDCP-related crypto helpers, RSA / signature plumbing
- `i2c_scdc.c`
  - DDC transfer engine and SCDC state machine
- `audio_earc.c`
  - eARC / ARC audio control, audio timing and state machines
- `hdmi_frl_video.c`
  - FRL video-side state machines, event flow, HPD / training integration
- `hw_misc.c`
  - platform glue, interrupts, peripheral control, clock / power / misc state helpers

## Reconstruction philosophy

The main intent of this tree is to preserve the original firmware logic in readable form.

That means the work is guided by these rules:

- preserve the original control flow when it can be recovered
- preserve argument meaning and side effects, not just high-level intent
- keep raw register and SRAM state visible where that is important to the logic
- prefer exact parser and state-machine behavior over "clean" abstractions
- preserve distinct failure paths when the original firmware distinguishes them

In practice, this code was rewritten by comparing reconstructed functions against the original monolithic decompilation function-by-function.

## Current status

Most of the firmware has been rewritten out of IDA-style autogenerated C into structured C.

Major modules are present and syntax-clean for the ARM target, but fidelity is not equally complete everywhere.

Broadly:

- system / scheduler / utility code is in good shape
- SCDC / DDC, audio, video, and platform code have been substantially reconstructed
- some crypto / IMG4 / HDCP paths are still only partial or conservative rather than exact

In particular, `crypto_hdcp.c` still contains the biggest remaining semantic gap versus the original firmware. The low-level parsing and tag checks are much closer now, but the recursive IMG4 / HDCP validation chain is still not a one-to-one reconstruction.

## Important caveats

- This is reconstructed code, not ground-truth source.
- "Readable" does not automatically mean "fully verified against the original."
- Some functions are intentionally conservative and fail closed instead of trying to guess missing crypto behavior.
- Some return-code behavior in parser-heavy paths may still differ from the original firmware.
- Host-toolchain warnings do not always reflect target behavior; this code is meant for an ARM freestanding environment, not a desktop build.

## How to verify syntax

This tree is intended to be syntax-checked as freestanding ARM Thumb code.

Example:

```bash
clang -fsyntax-only -std=c11 -I. \
  -target arm-none-eabi -mcpu=cortex-m4 -mthumb \
  -ffreestanding -nostdinc \
  -isystem "$(clang -print-resource-dir)/include" \
  system_main.c
```

To check all source files:

```bash
for f in *.c; do
  clang -fsyntax-only -std=c11 -I. \
    -target arm-none-eabi -mcpu=cortex-m4 -mthumb \
    -ffreestanding -nostdinc \
    -isystem "$(clang -print-resource-dir)/include" \
    "$f" || exit 1
done
```

## What is not included here

This repository only contains the reconstructed source tree.

The original monolithic decompilation, address-to-file mapping material, and other scratch reconstruction artifacts were used during the rewrite process but are not part of this repository.

## If you are continuing this work

The most useful way to improve this tree is not generic refactoring.

The right workflow is:

1. pick a reconstructed function
2. compare it directly against the original decompiled function it came from
3. verify argument meaning, descriptor layout, side effects, and return behavior
4. only then rename or simplify local structure if the behavior still matches

The highest-value remaining work is in the IMG4 / HDCP validation path in `crypto_hdcp.c`.
